> [!note]+ Reply — where should cleaning live?
>
> The frame "does input cleaning benefit summarization?" hides the real question. Cleaning ASR errors (`llama 270b` → `llama2 70b`) is **LLM reasoning**, not a deterministic transform — you can't regex it, you need context to know `270b` is wrong because the speaker is talking about Llama 2. That changes which options are actually viable:
>
> - **Inside extract** — still no. Extract is meant to be deterministic and idempotent (same yt-dlp + transcript-api call → same bytes every run, re-runnable with `--force`). An LLM step breaks that, and it entangles fetching with judgement. Not just a purity argument: if the cleaning model changes its mind next month, your "raw" file silently shifts and you've lost the ground truth.
> - **Separate cleaning skill with persisted clean file** — only worth it if more than one downstream consumer reads it (summarizer + quote extractor + search, etc.). Right now you have one: the summarizer. Two files also force you to answer "which is canonical?" forever, and the prompt-cache savings only materialize if the next skill actually starts its prompt with the same cleaned text.
> - **Inside summarization, implicit (default)** — the summarizer reads the transcript end-to-end anyway. It mentally corrects `llama 270b` as it writes the summary, and the corrected wording ends up in the output. No separate artifact, no extra tokens, no consistency question. Raw stays canonical.
> - **Inside summarization, explicit first pass** — only if you find the implicit version actually leaks ASR errors into summaries (likely on technical terms the model doesn't recognize as wrong). Then a small first-pass step inside `summarize-youtube`: "scan transcript for likely ASR errors using description / chapter titles / channel context as priors." Same skill, cached prompt, cheap.
>
> So your decision tree collapses: ship `summarize-youtube` with **no** cleaning step. If after a few real summaries you see ASR errors actually leak through, add an explicit first pass **inside that same skill** — not a separate skill, not a change to extract. The raw file stays the ground truth; the cleaned version only exists transiently inside the summarizer's reasoning, which is where it belongs.

> [!note]+ Reply — rest of summarization process (items 3 & 4)
>
> **3.1 — chapter quality survey:** Right instinct to look at the testset first. You can't design the chapter-handling logic without seeing the distribution. Concrete next step: skim each testset video and tag it `{good chapters, generic chapters, description-timestamps only, none}`. The shape of that distribution tells you whether to invest in sophisticated handling or just need one fallback path. Don't design before you have that shape.
>
> **3.2 — add a "meaningful chapter" flag to extract?** No — same trap as cleaning. Judgement would creep into the deterministic layer. Keep extract providing **facts** (`chapters`, `chapters_authoritative`, `has_real_chapters`, `has_key_moments`, raw `description`); let the summarizer make the **judgement**. To kill the "fear of not knowing what AI would do," require the summarizer to print one provenance line at the top of its output, e.g.:
> > `Segmentation: yt-dlp chapters (12 sections)`
> > `Segmentation: derived from description timestamps (yt-dlp chapters were generic)`
> > `Segmentation: AI-derived from transcript, no chapter source`
>
> Now AI's choice is auditable. You can spot-check across the testset and, if it consistently picks badly, you'll see it in the provenance line and tune the prompt.
>
> **3.3 — different effort per metadata quality:** doesn't need multiple skill modes. One prompt with a cascade works: "Use yt-dlp chapters if substantive section titles; else fall back to description timestamps; else derive segments from transcript content shifts." Same skill, AI follows the cascade, the provenance line above proves which branch fired.
>
> **3.4 — description as background, skip links/comments:** correct scope cut for v1. Worth being explicit in the `summarize-youtube` SKILL.md "When NOT to use" section so future-you doesn't get tempted to expand into link-following or comment-mining — those are separate skills if they're ever needed.
>
> **4 — output shape:**
> - **"Keep speaker's tone" vs summary** is a real tension. Pure summary compresses and loses voice; preserving voice pulls you toward structured re-transcript. Pick a primary: I'd make *understand-what-was-covered* primary (summary), tone-preservation secondary — verbatim quotes only where the phrasing itself matters (definitions, signature claims, a punchy one-liner).
> - **TOC vs chapter table:** the TOC is mostly redundant — Obsidian's outline panel auto-generates one from `##` headers. The chapter/segment table is more useful: `timestamp | title | one-line gist`, all in one glance for navigation. Recommend just the table, drop the separate TOC.
> - **Two-level hierarchy:** good principle, don't codify a hard rule like "subsections when chapter >2min." Let AI judge per video, with a constraint in the prompt: "use H3 only when a chapter has genuinely distinct sub-topics; otherwise keep flat." Combined with the provenance line, you'll see whether AI is over- or under-nesting and can adjust.
